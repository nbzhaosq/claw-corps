# 测试策略 - Claw Corps 项目

## 1. 测试目标

验证 Claw Corps（多智能体开发编排工具）的核心功能：
- **用户管理**：团队成员分配和管理
- **数据展示**：项目列表、状态展示、评估结果
- **简单统计**：功能点统计、复杂度评分、团队规模

## 2. 测试范围

### 2.1 功能测试（Functional Testing）

#### 用户管理功能
- ✅ 创建项目时初始化空团队
- ✅ 添加团队成员
- ✅ 分配角色和职责
- ✅ 查询团队成员信息

#### 数据展示功能
- ✅ 项目列表展示（带过滤）
- ✅ 项目详细信息展示
- ✅ 复杂度评估结果展示
- ✅ 进度追踪可视化

#### 简单统计功能
- ✅ 功能点数量统计
- ✅ 复杂度评分计算
- ✅ 团队规模统计
- ✅ 技术栈复杂度评估

### 2.2 非功能测试

#### 性能测试
- 大量项目列表加载性能
- 复杂度评估算法性能

#### 异常处理
- 无效项目ID处理
- 空数据处理
- 配置文件损坏恢复

## 3. 测试方法

### 3.1 单元测试
- **目标**：核心业务逻辑
- **框架**：Node.js 内置 test runner
- **覆盖**：
  - Storage 类的所有方法
  - ComplexityAssessor 类的评估算法
  - 数据验证和转换函数

### 3.2 集成测试
- **目标**：命令行交互流程
- **工具**：子进程执行 CLI 命令
- **场景**：
  - 完整的项目创建流程
  - 复杂度评估流程
  - 团队分配流程

### 3.3 端到端测试
- **目标**：真实用户场景
- **方法**：模拟完整工作流
- **用例**：
  - 从创建到执行的完整生命周期
  - 并发项目管理

## 4. 测试环境

- **Node.js**: >= 18.0.0
- **操作系统**: macOS / Linux
- **测试数据**: 隔离的 ~/.claw-corps-test 目录

## 5. 测试数据策略

### 5.1 测试数据隔离
- 使用独立的环境变量 `CLAW_CORPS_HOME`
- 每个测试用例前清理测试目录
- 测试后不污染真实数据

### 5.2 边界值数据
- 空项目列表
- 单个项目
- 大量项目（100+）
- 超长名称和描述

## 6. 测试执行计划

### Phase 1: 单元测试（优先级：高）
- [ ] Storage 类测试
- [ ] ComplexityAssessor 类测试
- [ ] 数据模型验证

### Phase 2: 集成测试（优先级：中）
- [ ] init 命令测试
- [ ] list 命令测试
- [ ] assess 命令测试
- [ ] status 命令测试

### Phase 3: 端到端测试（优先级：低）
- [ ] 完整工作流测试
- [ ] 并发场景测试

## 7. 验收标准

- 单元测试覆盖率 >= 80%
- 所有核心功能测试用例通过
- 无阻塞性缺陷
- 性能符合预期（列表加载 < 1s）

## 8. 风险与对策

### 风险1：文件系统依赖
- **影响**：测试可能污染真实数据
- **对策**：使用环境变量隔离测试目录

### 风险2：异步操作
- **影响**：测试可能出现竞态条件
- **对策**：使用 async/await 和适当的等待机制

### 风险3：交互式命令
- **影响**：inquirer 提示会阻塞测试
- **对策**：提供非交互模式选项
